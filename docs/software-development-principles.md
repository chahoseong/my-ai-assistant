# software-development-principles.md

실무에서 쓰이는 원칙을 정리한 목록이다. 프로젝트를 진행하며 사용자에게 가르치기 위해 작성했다.

## 일반 소프트웨어 엔지니어링 원칙

- **SOLID**: 단일 책임(SRP), 개방-폐쇄(OCP), 리스코프 치환(LSP), 인터페이스 분리(ISP), 의존성 역전(DIP)
- **DRY / KISS / YAGNI**: 중복 배제, 단순 유지, 필요할 때만 만들기
- **테스트 피라미드**: 빠른 단위 테스트를 두텁게, 통합 테스트는 얇게, E2E 테스트는 최소한으로
- **코드 리뷰**: 동작·엣지케이스·보안·성능 관점의 리뷰 습관
- **Trunk-based Development / CI-CD**: 작은 단위로 자주 통합하고, 항상 배포 가능한 상태를 유지

## 백엔드 엔지니어링 원칙

ISO/IEC 25010 소프트웨어 품질 모델에 근거한, 백엔드 시스템이 갖춰야 할 품질 목표.

- **확장성 (Scalability)**: 사용자와 데이터가 늘어나도 성능 저하 없이 대응하는 능력
  - 수평적 확장(scale-out): 서버 성능을 높이는 대신 대수를 늘려 부하 분산
  - 캐싱: 자주 조회되는 데이터를 메모리에 저장해 DB 접근 최소화
  - 비동기 처리: 오래 걸리는 작업은 큐로 넘겨 요청을 막지 않음

- **안정성 및 가용성 (Reliability & Availability)**: 장애가 발생해도 중단 없이 동작하거나 빠르게 복구되는 능력
  - 무상태성(Statelessness): 서버가 클라이언트 상태를 저장하지 않아 어떤 서버로 요청이 가도 동일하게 응답
  - 로드 밸런싱: 트래픽을 여러 서버로 분산
  - 장애 격리: 핵심 서비스와 부가 서비스를 분리해 장애 전파 방지
  - 이론적 근거: 분산 컴퓨팅의 8가지 오류(네트워크는 항상 신뢰할 수 있다는 식의 잘못된 가정을 경계), 멱등성(같은 요청을 여러 번 처리해도 결과가 동일해야 함), Postel의 법칙(보낼 때는 엄격하게, 받을 때는 관대하게)

- **유지보수성 (Maintainability)**: 코드와 시스템을 쉽게 이해·수정·확장할 수 있는 구조
  - 관심사 분리(SoC): 비즈니스 로직, 데이터 접근, API 라우팅 등의 역할을 명확히 분리
  - (SOLID, 테스트 피라미드는 위 일반 원칙 참고 — 여기서도 그대로 기여한다)

- **보안성 (Security)**: 인가되지 않은 접근과 데이터 유출로부터 시스템과 사용자 데이터를 보호
  - 최소 권한 원칙: 서버·DB·API에 필요한 최소한의 접근 권한만 부여
  - 데이터 암호화: 민감 정보는 해시로 저장, 전송 중 데이터는 암호화
  - 입력값 검증: 모든 사용자 입력을 검증해 SQL 인젝션·XSS 등 예방

- **관측가능성 (Observability)**: 시스템 내부 상태를 외부에서 파악하고 모니터링할 수 있는 체계
  - 로깅: 주요 이벤트와 에러 정보 기록
  - 메트릭: CPU/메모리/요청 수/응답 시간 등 수치화
  - 트레이싱: 하나의 요청이 여러 서비스를 거치는 과정 추적
