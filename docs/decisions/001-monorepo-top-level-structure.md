# ADR-001: 모노레포 최상위 구조로 `frontend/` + `backend/` 사용

## Status
Accepted

## Date
2026-07-15

## Context
저장소를 어떻게 나눌지 정해야 했다. 이 프로젝트는 혼자 진행하는 학습 프로젝트이고, 프론트엔드(TypeScript/React)와 백엔드(Python/FastAPI)로 구성된 폴리글랏 구조다. 먼저 모노레포/폴리레포 여부를, 그다음 모노레포 내부의 최상위 구조를 정해야 했다.

## Decision
하나의 저장소(모노레포) 안에 `frontend/`, `backend/` 디렉토리를 둔다. 이 구조는 프로젝트의 실제 모습(공유 코드 없는 두 개의 독립 애플리케이션)을 정직하게 반영하고, 각자 독립된 패키지 매니저(npm/uv)로 관리할 수 있게 한다.

## Alternatives Considered

### 폴리레포 (프론트엔드/백엔드를 별도 저장소로 분리)
- Pros: 프론트엔드와 백엔드를 독립적으로 배포·버전 관리할 수 있다. CI/CD가 각자 분리되어 깔끔하다.
- Cons: 이 이점들은 팀이 나뉘어 있거나 배포 주기가 서로 다를 때 의미가 있는데, 지금은 혼자 진행하는 프로젝트라 그런 필요가 없다. 오히려 저장소를 오가며 컨텍스트를 전환해야 하는 비용만 남는다.
- Rejected: 분리에 따른 이점을 누릴 조건(팀 분리, 독립 배포 주기)이 없고, 학습 프로젝트 특성상 전체 스택을 한눈에 보며 진행하는 게 더 유리하다.

### `apps/web` + `apps/api` + `packages/`
- Pros: Turborepo/Nx 같은 모노레포 도구와 자연스럽게 통합된다. 여러 앱이 코드를 공유할 때 `packages/`가 그 위치가 된다.
- Cons: 이 이점은 여러 앱이 **런타임 코드를 공유**할 때 성립하는데, 프론트(TypeScript)와 백엔드(Python)는 언어 자체가 달라 런타임 코드를 공유할 수 없다. Turborepo/Nx도 JS 생태계 도구라 uv로 관리하는 Python 쪽엔 적용되지 않는다.
- Rejected: 핵심 이점(공유 `packages/` + 모노레포 툴링)이 성립하지 않는 상황에서 이 컨벤션을 쓰면 "공유 패키지가 있는 척"하는 구조만 남는다.

## Consequences
- 프론트엔드와 백엔드는 같은 저장소 안에서 각자의 패키지 매니저와 독립적인 의존성 트리를 가진다.
- 나중에 정말 공유할 코드(예: API 타입 스키마)가 필요해지면, 그때 `packages/`를 추가하거나 OpenAPI 스펙 기반 코드 생성 같은 다른 방법을 고려한다.
- 팀이 나뉘거나 독립 배포가 필요해지는 시점이 오면, 이때 폴리레포 전환을 재검토한다.
