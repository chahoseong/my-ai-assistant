# ADR-002: 단일 프로세스 대화 잠금 유지

## Status
Accepted

## Date
2026-07-17

## Context
한 대화에서 응답을 생성하는 동안 같은 대화에 대한 두 번째 요청은 `409 Conflict`로 거절해야 한다. 현재 구현은 `asyncio.Lock`과 Python 메모리의 집합으로 대화 ID를 추적한다.

이 잠금은 하나의 FastAPI 프로세스 안에서만 공유된다. 여러 worker 또는 여러 애플리케이션 replica를 실행하면 각 프로세스가 서로 다른 잠금 집합을 가지므로, 같은 대화의 요청이 서로 다른 프로세스에서 동시에 처리될 수 있다. 각 데이터베이스 저장은 정상적으로 commit될 수 있어도, 사용자 메시지와 어시스턴트 응답의 턴 순서 및 LLM이 참조하는 대화 맥락이 어긋날 수 있다.

## Decision
현재는 단일 FastAPI worker를 전제로 프로세스 로컬 대화 잠금을 유지한다. 멀티워커 또는 다중 replica 배포는 이 잠금으로 보호되지 않는다는 점을 명시하고, 공유 잠금은 실제 확장 요구가 생길 때 도입한다.

## Alternatives Considered

### Redis 기반 분산 잠금
- Pros: 여러 worker와 여러 replica 사이에서 잠금 상태를 공유할 수 있고, 애플리케이션 서버를 수평 확장하기에 적합하다.
- Cons: Redis 운영, 연결 장애, 잠금 만료와 소유권 관리라는 새 운영·설계 복잡성이 생긴다.
- Rejected: 현재 단일 worker 개발 단계에는 해결하려는 문제보다 추가 복잡성이 크다.

### PostgreSQL advisory lock
- Pros: 이미 사용하는 PostgreSQL을 공유 조정자로 활용할 수 있어 별도 Redis가 필요 없다.
- Cons: 잠금이 DB 연결에 묶이므로 스트리밍 수명과 연결 수명, 장애 시 해제를 신중하게 설계해야 한다.
- Rejected: 현재 요구에는 과도하며, 멀티워커 배포를 도입할 때 실제 트래픽과 연결 전략을 바탕으로 다시 비교해야 한다.

### 한계를 기록하지 않음
- Pros: 추가 문서가 필요 없다.
- Cons: 운영자가 여러 worker에서도 `409` 보장이 유지된다고 오해할 수 있다.
- Rejected: 현재 보장 범위를 명시하지 않으면 확장 시 데이터 의미론이 깨질 위험을 발견하기 어렵다.

## Consequences
- 동일 대화의 중복 요청 방지와 `409 Conflict` 보장은 단일 FastAPI 프로세스에서만 유효하다.
- 서로 다른 대화는 기존처럼 동시에 스트리밍할 수 있다.
- 멀티워커 또는 다중 replica로 확장하기 전에 이 ADR을 재검토하고, Redis 또는 PostgreSQL advisory lock 기반의 공유 잠금으로 교체해야 한다.
