# ADR-003: 세션 쿠키 채택

## Status
Accepted

## Date
2026-07-18

## Context
사용자를 식별하고, 로그인 이후의 요청에서도 계속 그 사용자임을 확인할 방법이 필요했다. 브라우저 기반 프론트엔드(증분 5)가 이후에 붙을 예정이라, 클라이언트가 브라우저라는 조건도 고려해야 했다. 이 서비스는 단일 로컬 서버로 운영되며, 여러 서버 인스턴스나 여러 마이크로서비스가 인증 상태를 공유해야 하는 상황이 아니다.

## Decision
세션 쿠키를 사용한다. 서버가 무작위 세션 토큰을 발급해 `sessions` 테이블(PostgreSQL)에 사용자 ID와 만료 시각을 저장하고, 클라이언트에는 그 토큰만 httpOnly 쿠키로 전달한다. 로그아웃은 해당 세션 행을 삭제해 즉시 철회한다.

## Alternatives Considered

### JWT (httpOnly 쿠키 전달)
- Pros: 서버가 상태를 갖지 않아, 여러 서버 인스턴스나 마이크로서비스가 공유 저장소 없이 각자 토큰을 검증할 수 있다. 채용 공고에서 자주 언급되는 기술이라 그 자체로 학습 가치가 있었다.
- Cons: 이 무상태성의 이점은 여러 서버/서비스가 있을 때만 발휘되는데, 이 프로젝트는 단일 로컬 서버라 그 조건이 없다. 대신 무상태이기 때문에 만료 전 토큰을 철회할 방법이 없어, 짧은 만료 시간이나 리프레시 토큰 같은 보완 장치가 필요해진다.
- Rejected: 리프레시 토큰까지 추가하면 실제로 없는 요구(수평 확장) 때문에 복잡도만 늘리는 성급한 일반화가 된다. JWT의 실질적 이점이 발휘되는 상황(여러 서버·서비스, 모바일/제3자 클라이언트)은 이후 다른 프로젝트에서 다루기로 하고, 이번엔 아키텍처에 맞는 세션 쿠키로 되돌렸다.

### JWT + 리프레시 토큰
- Pros: 실무에서 JWT의 철회 불가 문제를 완화하는 표준 조합이다.
- Cons: 별도 토큰 종류, 안전한 보관, 회전 로직이 추가되는데, 애초에 무상태가 필요 없는 프로젝트에 그 복잡도를 감수할 이유가 없다.
- Rejected: 없는 요구를 위한 복잡도 추가.

## Consequences
- 로그아웃 시 서버 기록만 삭제하면 즉시 철회된다 — JWT가 가졌던 "만료 전까지 철회 불가" 문제가 사라진다.
- 서버가 상태(세션 테이블)를 가지므로, 여러 서버 인스턴스로 수평 확장하려면 세션 저장소를 공유해야 한다 (지금은 해당 없음).
- 세션 조회를 위해 매 요청마다 DB 조회가 한 번 추가된다.
- JWT 관련 지식(서명, 클레임 검증)은 이 프로젝트에서 학습하지 않으며, 그 무상태 이점이 실제로 필요한 다른 프로젝트에서 다루는 것으로 미룬다.
