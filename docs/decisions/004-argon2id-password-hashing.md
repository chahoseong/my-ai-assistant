# ADR-004: 비밀번호 해싱으로 Argon2id 채택

## Status
Accepted

## Date
2026-07-18

## Context
비밀번호를 저장할 해싱 방식을 정해야 했다. SHA-256 같은 일반 해시 함수는 계산이 빨라 오프라인 무차별 대입에 취약해 애초에 후보에서 제외했다. 비밀번호 전용 해시 함수 중에서 선택해야 했다.

## Decision
Argon2id를 채택한다(argon2-cffi). 파라미터는 라이브러리의 기본 프로필을 스펙에 고정해서 명시하고, 로그인 성공 시 `check_needs_rehash()`로 재해싱 필요 여부를 확인한다. hash/verify 연산은 비동기 이벤트 루프를 막지 않도록 threadpool에서 실행한다.

## Alternatives Considered

### bcrypt
- Pros: 25년 이상 검증된 알고리즘으로 여전히 안전하다고 평가된다. 생태계 지원이 넓다.
- Cons: 메모리 요구가 없어 GPU/ASIC 병렬 공격에 Argon2id보다 상대적으로 약하다. 72바이트를 초과하는 입력을 조용히 잘라버리는 특이 동작이 있다.
- Rejected: 새 프로젝트에서 bcrypt를 선택할 이유(레거시 호환)가 없고, OWASP가 1순위로 권고하는 Argon2id를 따르는 것이 학습·포트폴리오 관점에서 더 적절하다.

### SHA-256 (또는 다른 일반 해시 함수)
- Pros: 계산이 빠르고 널리 지원된다.
- Cons: 바로 그 "빠름"이 비밀번호 저장에는 치명적이다 — 탈취된 해시에 대한 오프라인 무차별 대입이 GPU로 초당 수십억 회 가능해, 낮은 엔트로피의 실제 비밀번호 대부분이 뚫린다. 솔트·메모리 요구 같은 완화 장치도 기본 내장되어 있지 않다.
- Rejected: 비밀번호 저장 용도로 사실상 부적합하다고 널리 알려져 있어 후보에서 제외했다.

## Consequences
- Argon2id 기본 프로필은 검증 1회당 약 64MiB 메모리를 요구한다 — 동시 로그인이 많아지면 서버 메모리 사용량이 비례해 늘어난다.
- hash/verify를 threadpool로 넘겨야 하며, 직접 이벤트 루프에서 호출하면 다른 SSE 요청이 지연될 수 있다.
- 향후 argon2-cffi의 기본 파라미터가 바뀌어도 `check_needs_rehash()`로 기존 사용자 해시를 점진적으로 업그레이드할 수 있다.
